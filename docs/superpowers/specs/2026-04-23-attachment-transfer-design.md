# Attachment Transfer Design

**Problem**

当前 `chatbox` 只支持纯文本聊天。用户已经明确希望补齐附件能力，尤其是：

- 图片发送与查看
- GIF 发送
- 任意文件附件
- 路由器 host 临时托管附件，便于多端按需下载

现有实现不适合直接承载大文件：

- 聊天协议当前只有单条 UTF-8 文本消息
- `session.Message.Body` 不适合承载二进制
- 房间广播与 transcript 都围绕“文本消息 + 少量控制消息”设计
- 用户已明确暂时不优先做复杂可靠性能力，但希望第一版能支持最大 `2GB` 的附件

**Goal**

在保持当前聊天主链路简单可用的前提下，为 `chatbox` 增加“附件上传到 host、聊天室广播附件消息、其他客户端按需下载并本地打开”的能力。

第一版目标如下：

- 支持图片和任意文件附件，共用同一套机制
- 支持单文件最大 `2GB`
- host 只保存密文，不保存明文
- 附件由发送方先上传成功，再向聊天室发送附件消息
- 接收方默认不自动下载，只在用户手动打开或下载时再拉取
- 路由器 host 自动清理超过 `7` 天的附件
- 兼容老版本客户端，至少保证“不会挂掉、能看到一条可读文本”

## Architecture

第一版采用“聊天 TCP 通道 + 独立附件 HTTP 服务”双通道方案。

- 聊天消息继续沿用现有 TCP 会话与房间广播逻辑
- `host` 额外启动一个附件 HTTP 服务，默认监听 `chat-port + 1`
- 发送方上传附件时，文件内容不进入聊天室主连接
- 上传成功后，发送方再发出一条普通聊天消息，表达“这里有一个附件”
- 接收方看到附件消息后，可手动执行下载或打开

这样可以把“大文件传输”从现有文本聊天协议里剥离出去，避免：

- 2GB 文件拖垮聊天主连接
- 在 `session` 和 `room` 里重做复杂分片、流控与恢复机制
- transcript 结构被二进制内容污染

## Attachment Service

`host` 增加一个最小附件服务，仅用于附件上传、元信息查询、密文下载和删除。

建议接口：

- `POST /v1/attachments`
- `GET /v1/attachments/{id}/meta`
- `GET /v1/attachments/{id}/blob`
- `DELETE /v1/attachments/{id}`

约束：

- 全部使用流式 I/O
- 不把整个文件一次性读入内存
- 服务只在 `host` 端运行，`join` 不开启
- 默认为聊天监听端口加一，例如：
  - 聊天 `7331`
  - 附件服务 `7332`

## Encryption And Authentication

第一版继续只依赖聊天 `PSK`，不引入第二套口令或账号体系。

客户端基于 `PSK` 派生两类密钥：

- 附件内容加密 key
- 附件服务请求签名 key

文件内容规则：

- 发送方在本地加密后再上传
- host 只保存密文 blob
- 下载方拉取密文后在本地解密

请求鉴权规则：

- 上传、查询、下载、删除都带时间戳和 HMAC 签名
- host 用同一 `PSK` 派生校验 key 验签
- 未持有正确 `PSK` 的客户端即使知道附件 ID 也无法访问

第一版不做用户级 ACL。只要属于同一个房间并持有 `PSK`，就可以访问该房间附件。这与当前聊天室的信任模型一致。

## Attachment Message Format

附件消息不使用隐藏控制帧，而是使用“可见、可读、可解析”的普通 UTF-8 文本消息。

设计目标：

- 新版本客户端可将其识别并渲染为附件条目
- 老版本客户端不会崩，也不会看到难以理解的二进制或控制串

附件消息至少应表达：

- 附件类型：`image` 或 `file`
- 文件名
- 文件大小
- 附件 ID

新版本客户端行为：

- 解析该消息并按附件条目渲染
- 展示短附件 ID，便于 `/open` 和 `/download`

老版本客户端行为：

- 只会看到一条可读文本，例如：
  - `shared image: cat.gif (12.3 MB) #a1b2c3`
  - `shared file: report.pdf (4.8 MB) #d4e5f6`

下载与解密所需的完整元信息不直接放进聊天正文，而是通过附件 ID 向 host 查询。

## CLI And TUI Interaction

第一版尽量复用现有命令和 TUI 交互，不增加新的全局模式。

发送附件：

- 新增 `/attach <path>`
- 图片、GIF、普通文件都共用该命令
- 客户端根据扩展名和 MIME 自动标记 `image` 或 `file`
- macOS 用户可在 `/attach ` 后直接拖文件路径到终端

本地校验：

- 路径存在
- 是普通文件
- 文件大小不超过 `2GB`

聊天里展示：

- 上传成功后只生成一条附件消息
- 渲染类似：
  - `[image] cat.gif (12.3 MB) #a1b2c3`
  - `[file] report.pdf (4.8 MB) #d4e5f6`
- 已下载、已过期、不可用等状态只作为简短本地状态附加，不额外写 system 行

命令：

- `/open <attachment-id>`：按需下载到本地缓存并直接打开
- `/download <attachment-id> [dest]`：下载到指定路径；不传时使用默认缓存目录
- `/help` 和 TUI 的 `/` 联想中补充命令说明

TUI 交互：

- 继续复用 `copy mode`
- 当选中的是附件消息时：
  - `o`：下载并打开
  - `d`：仅下载
  - `y`：保持复制文本能力
- 输入区 hint 根据当前选中消息动态变化

打开方式：

- macOS：`open`
- Linux：`xdg-open`
- Android / Termux：优先 `termux-open`，缺失时只打印本地路径

## Upload And Download Progress

上传和下载都需要本地可视化进度，但不进入聊天历史。

TUI：

- 在底部状态区显示实时进度，例如：
  - `uploading cat.gif 37% 742MB/2.0GB 18.4MB/s ETA 01:12`
- 完成或失败后，状态区恢复常规内容

`scrollback`：

- 使用输入区附近的单行临时刷新显示进度
- 不写入终端历史，不刷出多条百分比日志
- 完成后恢复正常输入提示

关键约束：

- 上传未完成前，不向聊天室发送附件消息
- 上传失败时，只提示本地错误
- 下载失败时，只提示本地错误，用户可以再次执行 `/open` 或 `/download`

## Router Storage

host 本地目录建议拆分为：

- `attachments/blobs/`
- `attachments/meta/`

元信息应至少包含：

- 附件 ID
- 房间 key
- 发送者名称
- 发送者 identity
- 原始文件名
- 附件类型
- 文件大小
- 内容摘要
- 上传时间
- 过期时间
- 撤回状态

host 不保存聊天记录副本，只保存附件所需的最小元信息和密文文件。

## Lifecycle And Revoke

默认附件 TTL 固定为 `7` 天。

清理策略：

- host 启动时先执行一次清理
- headless host 内部再启动每日清理任务
- 清理时删除已过期的元信息和对应 blob
- 删除逻辑必须幂等，允许元信息和 blob 其中一方先缺失

撤回策略：

- 附件消息一旦被撤回：
  - 聊天记录沿用现有撤回显示逻辑
  - host 同时删除对应附件
- 已过期、已撤回或已损坏的附件，再次打开时应返回明确错误

上传后发送消息失败的处理：

- 如果附件上传成功但聊天室消息发送失败，客户端应立即尝试删除该附件
- 删除失败仅记录本地错误，不再重试成后台任务

## Compatibility

第一版必须保证以下兼容边界：

- 老版本客户端不会因为附件消息直接崩溃
- 老版本客户端至少能看到一条可读文本
- host 和 join 不要求先引入“全量新协议分片能力”

这意味着第一版的附件消息正文必须：

- 保持 UTF-8 文本
- 对未知客户端是“普通聊天文本”
- 对新客户端是“可结构化解析的附件消息”

## Error Handling

- `/attach` 指向不存在路径：显示本地错误
- `/attach` 指向目录或非常规文件：显示本地错误
- 文件超过 `2GB`：显示本地错误
- 上传鉴权失败：显示本地错误，不发聊天消息
- 下载鉴权失败：显示本地错误，不修改聊天历史
- 元信息存在但 blob 缺失：显示附件已损坏或已清理
- 已过期或已撤回附件：显示附件已不可用
- 打开命令不存在：显示本地保存路径

所有这些错误都不应污染远端聊天记录，也不应变成聊天室里的 system message。

## Testing

必须覆盖以下行为：

- `/attach` 成功上传后只发送一条附件消息
- 上传失败时不会产生聊天室消息
- 生成的附件消息对老版本客户端是可读文本
- 新版本客户端可正确解析 `image` 与 `file` 两类附件
- `/open` 可触发按需下载、解密和本地打开
- `/download` 可下载到默认目录和显式目标目录
- TUI 选中附件消息时 `o` / `d` 行为正确
- 上传进度和下载进度只显示为本地临时 UI，不进入历史
- host 只保存密文，不出现明文正文
- 超过 `7` 天的附件会被清理
- 附件撤回时，host 侧对应文件被删除
- 上传成功但聊天消息发送失败时，会尝试清理孤儿附件
- blob 缺失、签名错误、已过期附件都返回明确错误

## Non-Goals

- 断点续传
- 多并发分块加速
- 秒传或全局去重存储
- 路由器侧附件浏览 UI
- 聊天窗口内联图片渲染
- 自动下载所有附件
- 用户级权限控制
- 附件持久保留超过 `7` 天
