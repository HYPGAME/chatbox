# TUI Reply Card Rendering Design

## Context

当前 TUI 已经完成了第一阶段的引用优化：

- `quote` 不再把原文直接塞回输入框
- 输入区使用本地 `reply draft` 展示回复目标
- 发送时继续落成兼容旧版本的普通文本：
  - 第一行 `> sender [HH:MM] summary`
  - 第二行开始为正文

这解决了输入区编辑问题，但发送后的展示仍然偏“技术文本”。

当前效果更像：

```text
[15:04] bob: reply alice [11:22] hello world...
收到，我晚点处理
```

它虽然已经比原始多行 `>` 更紧凑，但视觉层次仍然不够像“回复卡片”，在长会话里辨识度偏弱。

## Goal

在不升级协议、不修改历史存储结构的前提下，把“发送后引用消息”的 TUI 展示升级成更明确的小卡片引用样式。

目标：

- 回复消息在聊天区更容易被一眼识别
- 保持窄终端下的可读性
- 不破坏老格式引用与普通文本消息
- 不改变发送协议与同步协议

## Non-Goals

- 不引入结构化 `reply_to_message_id`
- 不支持点击引用跳转
- 不识别并强转旧多行 `>` 引用
- 不修改 scrollback 模式的纯文本输出策略
- 不增加 hover、鼠标交互、上下文菜单等附加行为

## Chosen Direction

用户已确认以下展示方向：

- 使用“小卡片引用”样式
- 卡片内显示：
  - 发送人
  - 时间
  - 摘要
- 卡片位置：
  - 在上
  - 正文在下
- 卡片高度：
  - 双行
- 旧格式兼容策略：
  - 只处理新格式引用
  - 老的多行 `>` 引用保持原样

## Rendered Layout

回复消息在 TUI 中渲染为三段：

1. 主消息头：保留现有格式
2. 引用卡片：紧接在主消息头下方
3. 正文：卡片下方直接显示，不额外插空行

目标效果：

```text
[15:04] bob:
  │ alice · 11:22
  │ hello world...
收到，我晚点处理
```

其中：

- 第一行仍保留时间戳和当前消息发送人
- 引用卡片整体缩进 2 个空格
- 卡片使用 1 列左侧强调条
- 卡片固定两行：
  - 第一行 `sender · HH:MM`
  - 第二行 `summary`
- 正文从下一行开始，与主消息左侧正文起点对齐

## Visual Style

卡片样式采用“轻卡片”，不使用完整边框。

原因：

- Terminal.app 宽度有限，完整框线会显得拥挤
- 卡片内容经常跟随窄宽度换行，弱边框更稳
- 左侧细竖线更接近聊天应用里的“引用块”感受

具体建议：

- 卡片左侧使用细竖线，例如 `│`
- `sender · time` 使用较淡色
- `summary` 比第一行略亮，但仍低于正文权重
- 卡片背景不做重色块，避免和已存在的 sender colors、系统行、错误行冲突
- 卡片与正文之间不额外插入空白行

## Detection Rules

只识别当前这类“新格式引用”：

```text
> sender [HH:MM] summary
body...
```

识别条件：

- 第一行必须匹配 `> <sender> [HH:MM] <summary>`
- 必须存在第二行及之后的正文
- 第二行开始按正文处理，不继续解释为卡片内容

不匹配时：

- 直接按普通消息正文渲染

这意味着：

- 老多行 `>` 引用不会被错误折叠成卡片
- 用户手工输入的普通 `>` 文本只有在精确命中新格式时才会变成卡片

## Legacy Compatibility

旧格式多行引用示例：

```text
> alice [11:22]
> hello world
收到，我晚点处理
```

对此保持现状：

- 不尝试识别为卡片
- 按普通多行文本输出

这样可以避免误判和兼容风险。

## Attachment And Revoked Reply Cards

卡片摘要直接复用当前 reply draft 已生成的摘要文本。

因此发送后卡片天然支持：

- 图片引用：
  - `alice · 11:22`
  - `[图片] cat.gif`
- 文件引用：
  - `alice · 11:22`
  - `[文件] demo.zip`
- 撤回引用：
  - `alice · 11:22`
  - `已撤回一条消息`

不需要额外解析附件协议正文或撤回状态。

## Message Header Relationship

主消息头与卡片的关系需要保持清晰：

- 主消息头仍表示“当前这条消息是谁发的”
- 卡片只表示“它引用了谁的哪条消息”

因此不要把当前消息头和卡片压到同一行。

不采用这种形式：

```text
[15:04] bob: [reply alice · 11:22] hello...
```

因为它会让当前消息和被引用消息混成一层。

## Scrollback Strategy

scrollback 模式继续保留当前纯文本输出，不同步引入卡片化渲染。

原因：

- scrollback 依赖终端自带历史缓冲
- 该模式更适合保持稳定、可复制、可搜索的普通文本格式
- 卡片化更适合作为 TUI 视图增强，而不是 scrollback 的强约束格式

## Error Handling

- 如果识别出新格式引用但正文为空：
  - 不渲染卡片，回退为普通文本
- 如果消息本身已撤回：
  - 整条消息仍优先显示 `已撤回一条消息`
  - 不展示原回复卡片内容
- 如果摘要过长：
  - 继续使用发送前已截断的摘要
  - 渲染层不再做二次复杂裁剪逻辑

## Testing

需要覆盖以下测试：

1. 新格式引用消息在 TUI 中渲染为“小卡片 + 正文”
2. 卡片固定两行：
   - 第一行 `sender · HH:MM`
   - 第二行 `summary`
3. 卡片位于正文上方，且正文不额外多空行
4. 老格式多行 `>` 引用保持原样显示
5. 普通消息不受影响
6. 附件引用渲染为附件摘要卡片
7. 撤回引用渲染为撤回摘要卡片
8. 消息本身被撤回时，不再泄露原引用内容

## Future Work

如果后续要继续往“聊天 App 风格回复”靠近，再考虑：

- 点击引用跳转原消息
- 引用目标高亮定位
- 结构化 `reply_to_message_id`
- 卡片内显示附件类型图标或更丰富元信息

这些都应放到下一阶段，不与当前纯渲染升级绑定。
